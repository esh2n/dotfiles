# 日本語の「AIっぽさ」— パターン集

日本語圏には査読論文レベルの定量分析はほぼ存在せず、以下は実務者（ライター・編集者・
エンジニア）の**観察知**が中心。出典の種類を各所に明記する（技術記事 / SNS / 企業ブログ）。
最上位シグナルは複数ソースで一致して **①記号の癖 ②文末の均一 ③一次体験ゼロの一般論
④温度が一定** の4点。

## 1. 記号・表記の癖（一目で見抜かれる最上位）

- **太字（`**`）の乱用**。段落ごとに強調が出るのはテンプレ感。マークダウンのバグで `*` が本文に露出することも（句読点隣接時）
- **インラインコード（`` ` ``）の乱用**。関数名・コマンドなど識別子に使うのは正当だが、「重要」「ポイント」のような一般語や強調にまで囲むとAI臭。強調が太字＋コードの二重掛けになりがち
- **ダッシュ「——」「──」** での言い換え・挿入。英語の em dash 直輸入
- **コロン「：」の多用**：「ポイントは以下のとおりです：」は日本語として不自然。後ろに半角スペースが入る
- **絵文字付き見出し**（✅ 💡 🚀 ✨）、絵文字マーカー箇条書き
- **ダブルクォート「"　"」の過剰**、「」の中に『』の入れ子
- **矢印での構造化列挙**：「入力→検証→変換→保存→通知」のように処理順を記号でパッキングする。図解のつもりでも文章としてはAI臭が強い
- **スラッシュ「／」で概念並列**、「（）」補足のしすぎ
- **句読点が完璧すぎる**（誤字脱字ゼロも逆に不自然）
- **漢語の詰め込み**：「包括的視点から最適化された網羅的な情報提供」

出典: もとやま「AIっぽい文章表現大全」(note, 技術記事), こん太ぱぱ(note, マークダウン装飾バグ),
Togetter 大喜利まとめ(SNS), 上村菜穂(note, 実務観察)

**直し方:** 太字・絵文字見出しは原則削除。「——」は読点・句点・括弧に置換。「：」は「〜は次のとおり。」等の和文に。装飾ではなく文で強調する。バッククォートは識別子だけに絞る。

**AI臭ではないもの（誤検出注意）:** 英数字の左右の半角スペース（「CSV 取り込み」「E2E で」）は、和欧混植の**正統な組版ルール**（四分アキ）でありAI臭ではない。textlint の `ja-space-between-half-and-full-width` が統一を扱う対象。問題になるのは**入れる/入れないの不統一**だけ。一貫していれば著者のスタイルとして触らない。

## 2. 文末・リズムの癖

文末は**両極**がAI臭。均一な「です」連続も、体言止めの羅列も、どちらも機械的に映る。

- **文末の均一化**：「〜です。〜です。〜です。〜でしょう。」と同じ語尾が連続、抑揚がない（最頻出の指摘）
- **地の文での体言止め＋断定**（見落とされがち・重要）：地の文（散文）で「〜まで待つポーリングだ」のように名詞句で止めて「だ」で締めると芝居がかる。均一を崩すつもりで体言止めに寄せると**逆にAI臭が増す**。※ **箇条書き項目**の体言止めは自然（むしろ推奨）。NGなのは地の文での多用・断定
- 段落の閉じ方が毎回同じ（「以上がポイントです。次に進みましょう。」）

出典: もとやま(note), 上村菜穂(note), mieru-ca(技術記事) — 複数で最も繰り返し指摘。体言止め羅列は実使用フィードバックによる追記

**直し方:** 語尾を混ぜるが、**体言止めに逃げない**。基本は自然な動詞終わり（「〜した」「〜する」）に戻し、文の長短でリズムを作る。体言止めは1〜2箇所の効かせどころだけ。1段落に同じ語尾を3連続させない。

## 3. 定型的な言い回し

**文末・締め:**
「〜が重要です」「〜することが重要です」「〜と言えるでしょう」「〜と考えられます」
「〜という見方もできます」「〜することが望ましいでしょう」「いかがでしたか」「まとめると」「結論として」

**前置き・導入:**
「ご質問ありがとうございます」「結論から言うと」「ざっくり言うと」（そのあと結論が薄い）
「以下の3つの観点から説明します」「順に見ていきましょう」（構造を何度も宣言）

**保険・事なかれ（中立を装う）:**
「一概には言えませんが」「一般的には」「ケースバイケースです」「場合によります」
「メリットもあればデメリットもあります」「賛否が分かれるテーマです」

**根拠なき評価語:**
「非常に有効です」「大きなメリットがあります」「わかりやすいです」「妥当です」

出典: もとやま「AIっぽい文章表現大全」(note, 最も網羅的な原典), yosca(技術記事), カッパログ「チャッピー構文」(技術記事, 3年分履歴分析)

**直し方:** 締めの一般論は具体的な一言に置換。「〜と言えるでしょう」→ 断定するか、具体例で示す。前置きの定型は全削除して本題から始める。中立の保険は立場を取る形に。

## 4. 接続・構成の癖

- 接続詞の機械的多用：「さらに」「また」「したがって」「そのため」を文頭に置きすぎ
- バランス取り接続詞：「ただし」「一方で」「とはいえ」で中立を装う
- 三段構成テンプレ（総論→各論→まとめ）、「○○とは何か」「○○する3つの方法」の見出し型
- 短い内容でもすぐ見出し・箇条書き・表にする（形式の乱用）
- 流れの途中で唐突に箇条書き・整理が始まりリズムが変わる
- **列挙・手順を無理に散文化する**（重要）：本来 `1. 2. 3.` で書ける処理内容や実行手順を、「〜を作り、〜を用意し、〜してから〜する」と読点で繋いで1文に詰め込む。AIが「ちゃんとした文章」にしようとする典型。エンジニアのメモ・PR説明はむしろ箇条書きで淡々と書く

出典: もとやま(note), yosca(技術記事), Togetter(SNS)。散文化の指摘は実使用フィードバック

**直し方:** 接続詞を減らし、文の論理で繋ぐ。三段テンプレを崩す。短い内容は地の文で書く。**逆に、列挙・手順は散文に流さず素直に番号/箇条書きにする** — 記号パッキング（`A→B→C`）はNGだが、素の番号リストは自然。

## 5. トーン・内容の癖

- **一次体験・具体性の欠如**：「一般的には○○です」の他人事口調。体験談・数値・固有事例がない（最重要視）
- **抽象語・万能語**：「本質を押さえ、最適化し、価値を最大化する」「サポート」「提供」「調整」
- **温度が一定**：全体で熱量が変わらない、書き手の感情が見えない
- **平板な感情語**：「とても感動しました」「心から嬉しく思います」など一般的形容詞のみ
- **過剰に持ち上げる**、共感の水増し
- **完璧すぎる論理**：迷い・例外・言いよどみがない
- **不要な強調語**：「かなり」「いちばん」「圧倒的に」「本質的に」「まさに」

出典: malna(企業ブログ, 分析寄り), 上村菜穂(note), Togetter(SNS), カッパログ(技術記事)

**直し方:** 一般論を具体に。ただし**事実の捏造は禁止** — 体験談がないなら無理に作らず、抽象度を保つか著者に確認する。抽象動詞を具体動作に。強調語を削り、必要な一箇所だけ残す。

## 6. 翻訳調・不自然な語彙

- 英語直訳：`provide`→「〜を提供します」、`enable`→「〜を可能にします」の機械的直訳
- カタカナ抽象語の乱用：「型」「設計」「解像度」「粒度」「フレームワーク」「効きます」（＝「チャッピー構文」）
- 助詞の微妙なずれ：「初心者に向きます」（正: 「向いています」）
- 過度にフォーマル・回りくどい：「この件につきまして、ご対応のほど何卒よろしくお願い申し上げます」
- 敬体と常体の混在、主語の過剰明示 or 完全不在の両極

出典: カッパログ「チャッピー構文」(技術記事, Before/After有), AI翻訳の直訳調記事群, KDDI biz(企業)

**直し方:** 直訳語を和文の自然な動詞に。「〜を提供します」→「〜できる」「〜がある」。助詞を検算。敬体/常体を統一。

## 7. 圧縮構文・キメ文・用語のズレ（技術文書で特に重要）

実使用フィードバックによる追記。技術文書・PR説明・ドキュメントで顕著に出る癖。

- **倒置のキメ文**：「〜を保証するためだ」「〜を実現するためである」と、`保証する`/`実現する` のようなAI好みの動詞を倒置で締めて芝居がからせる。1文で決めにいく構文がわざとらしい
- **連体修飾の畳み掛け（過圧縮）**：「先行タスクの変更を前提とした暫定対応」「一定間隔で状態を確認しながら完了を待つ処理」のように、修飾を詰め込んで名詞1個に着地させる。情報密度は高いが人間はここまで圧縮しない
- **万能動詞での締め**：「〜を保証する」「〜を可能にする」「〜を実現する」「〜に対応する」。中身の動作を抽象動詞で丸めてしまう
- **曖昧な強調副詞**：「きっちり」「しっかり」「きちんと」「適切に」「厳密に」。数値も基準もなく強調だけする（「きっちり合わせてある」→ 何をどう合わせたのか中身がない）
- **対句「〜だけでなく〜も」**：2項目を対句で整えて“網羅した感”を出す。素直に2つ挙げれば足りる
- **括弧での補足の多用**：「空振り（何も取り込めていない）」のように、地の文に括弧で言い換え・補足を差し込む
- **自明な理由の「〜ため。」文**：「ずれると弾かれてしまうため。」のように、読めば分かる理由をわざわざ付け足す。そもそも不要な情報のことも多い（削る判断も含める）
- **用語のハルシネーション（最重要）**：AIはプロジェクト固有の呼称を知らないので、一般的で"それっぽい"専門用語を勝手に当てる（例：コードで「統合」と呼ぶ処理に「名寄せ」のような別分野の定番語を当てる、実在しない設定フラグや接尾辞をでっち上げる等）。読み手には「知ってるはずの用語がなんか違う」という違和感になる
  - **派手な造語だけではない（重要）**：「契約」「設定」「型」「ジョブ」のような**ごく普通の名詞**も、実コードの識別子・ドメイン呼称とズレる（コードでは別名で呼ぶ、そもそもそういう括り方をしていない等）。AIには正しい呼称を判別できない。**目立つ用語にだけ印を付けて安心せず、ドメイン名詞は総じて著者に委ねる**

**直し方:**
- キメ文の倒置をほどく：「〜を保証するためだ」→「〜だからだ」「〜に気づけないので」など、素の理由文に
- 過圧縮をほどく：修飾を1〜2文に割る。「先に入れた変更に依存しているので、その上に積む形にした」
- 万能動詞を具体動作に：「完了を保証する」→「終わったことを確認する」
- **用語は勝手に確定・変更しない**：正しい呼称を判断できない語は、原文のまま残して 〔※用語要確認〕 と印を付け、著者に委ねる。generic な言い換えに逃げるのが最悪手

## Before/After 実例

- 「この施策が有効です」→「この方法、実際に試したら予想以上によかった」（上村菜穂）
- 「近年、多くの企業で業務効率化が求められています。しかし、リソース不足でなかなか進みません。」→「『毎日残業で本来やるべき仕事に集中できない』——深刻な人手不足の中、こう悩む現場リーダーは少なくない。」（yosca プロ編集者リライト）
- 「初心者に向きます」→「初心者に向いています」（カッパログ、助詞）
